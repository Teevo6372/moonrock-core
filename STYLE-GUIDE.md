# Moonrock Documentation Style Guide

## Purpose

This guide establishes the writing, formatting, and documentation standards for every production document within Moonrock Core.

The goal is consistency.

Every document should read as though it was written by a single product team.

---

# Documentation Philosophy

Documentation is a product.

Every document should:

- Support implementation
- Reduce ambiguity
- Improve decision making
- Be understandable without chat history
- Scale as Moonrock grows

Write for the next developer, the next AI agent, and your future self.

---

# Voice & Tone

Moonrock documentation should be:

- Professional
- Clear
- Confident
- Helpful
- Practical
- Outcome focused

Avoid being:

- Overly technical
- Overly promotional
- Corporate
- Vague
- Wordy

---

# Writing Standards

Use plain English.

Keep paragraphs short.

Write one idea at a time.

Explain outcomes before implementation.

Prefer clarity over cleverness.

Avoid unnecessary buzzwords.

---

# Heading Structure

Use a logical hierarchy.

```text
H1  Document Title

H2  Major Section

H3  Subsection

H4  Supporting Detail (only when necessary)
```

Never skip heading levels.

Only one H1 should exist per document.

---

# Standard Section Order

When applicable, production documents should follow this order:

1. Purpose
2. Business Objective
3. Emotional Objective
4. Requirements
5. Layout or Process
6. Content
7. Technical Requirements
8. Accessibility
9. Acceptance Criteria
10. Revision Information

Not every document requires every section.

Choose the structure that best supports implementation.

---

# Terminology

Use approved Moonrock terminology consistently.

Approved terms include:

- Moonrock Flight Plan™
- Moonrock Launch Assessment
- Moonrock Growth Assessment
- Growth Hub
- Nova
- Virtual Growth Advisor
- Business Growth Partner

Do not introduce alternate names without approval.

---

# Requirement Language

Preferred wording:

- Must
- Should
- May

Avoid:

- Maybe
- Probably
- Try to
- Hopefully

Requirements should be measurable whenever possible.

---

# Lists

Use bullet lists for:

- Requirements
- Features
- Standards
- Deliverables

Use numbered lists for:

- Processes
- Workflows
- Sequential tasks

Keep lists concise.

---

# Formatting

Use:

- Consistent spacing
- Short paragraphs
- Descriptive headings
- Code blocks for filenames and commands

Avoid:

- Excessive bold text
- Large walls of text
- Inconsistent indentation

---

# Acceptance Criteria

Production specifications should conclude with measurable outcomes.

Example:

- Visitor understands the next step.
- Navigation remains clear.
- Mobile layout functions correctly.
- Accessibility requirements are satisfied.

Acceptance criteria should describe success, not effort.

---

# Version Control

Every production document should include:

- Version
- Status
- Owner
- Repository
- Last Updated

Changes to locked documents require a new version.

---

# Repository Standard

One topic.

One document.

One source of truth.

Avoid duplicate documentation.

Avoid conflicting instructions.

---

# Definition of Done

A document is complete when:

- It fulfills its purpose.
- It is implementation-ready.
- It has been reviewed.
- It has been approved.
- It has been committed to GitHub.
- It can stand alone without additional explanation.
